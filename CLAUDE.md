# TomorrowsMeal 開発ガイド & デザインルール

このファイルは、本プロジェクト（TomorrowsMeal webapp）における共通のデザインルール、テックスタック、および開発コマンドを定義したものです。

---

## 1. テックスタック・前提条件
- **CSS フレームワーク**: Tailwind CSS
- **UI コンポーネントライブラリ**: daisyUI
- **レスポンシブ対応**: 完全モバイルファースト
  - スマートフォン画面（幅: 375px〜420px程度）で最適に表示されること。
  - PCで見ても、画面中央にスマートフォンサイズで綺麗に収まる構成（例: `max-w-md w-full` を使用したセンタリング）にすること。

---

## 2. デザインコンセプト & カラーテーマ
- **アプリの世界観**: 「優しくて親しみやすい」「清潔感がある」「冷蔵庫・料理のワクワク感」
- **daisyUI テーマ**: `emerald`（または `cupcake`）をベースとする。
  - **プライマリーカラー (緑・エメラルド系)**: 新鮮な野菜、料理、健康をイメージ。
  - **セカンダリー / アクセントカラー**: 暖色系、料理の美味しさを引き立てるポップな色。
  - **背景色**: 清潔感のある明るい白・オフホワイトベース（`bg-base-200` や `bg-base-100` 等）。

---

## 3. 共通UIコンポーネントルール
*※原則として、daisyUIの標準クラスを最大限に活かし、独自のカスタムCSS（style.cssへの個別記述など）は書かないこと。*

### ボタン (`.btn`)
- アクションボタンは丸みを持たせ、`.btn-primary` または `.btn-secondary` を使用。
- 押しやすさを重視し、モバイルでタップしやすい十分なサイズ（最小でも高さ48px相当、例: `h-12` や `py-3`）を確保すること。

### カード (`.card`)
- 食材一覧、レシピ表示、および各種選択オプションは `.card .card-bordered .bg-base-100 .shadow-xl`（または `shadow-sm`）を使用し、適度な影と丸み（`.rounded-box` または `.rounded-2xl`）を持たせて浮き上がらせること。

### タイポグラフィ
- 日本語が美しく読みやすいフォント定義（例: `Plus Jakarta Sans` や `Noto Sans JP` など、OS依存のメイリオ等を避ける設定）を使用。
- 料理名や重要なヘッダーは `.text-xl .font-bold .text-base-content` で強調。

### 余白と配置
- スマートフォンの端でコンテンツが窮屈にならないよう、全体のコンテナには `.p-4` または `.p-6` の適切なパディングを設定すること。
- 縦方向の要素の並びは `.space-y-4`（または `.space-y-6`）を用いて一貫したディスタンスを保つこと。

---

## 4. 演出・UXルール
- **ローディング**: AI生成中などの待機画面では、daisyUIの `.loading .loading-dots .loading-lg` を使用し、「AIが献立を考えています...」といったポップで遊び心のある文言を添えること。
- **状態の表現**: 賞味期限切れやエラー、警告などのバッジ・テキストには `.text-error` や `.badge-error` を使用し、視覚的に一瞬で伝わるようにすること。

## 5. 主要な開発コマンド
### アプリケーションのビルド・起動
```bash
# Dockerコンテナのビルドおよびバックグラウンド起動
docker compose up --build -d

# コンテナの停止
docker compose down
```

### API の確認 (curl による疎通確認)
```bash
# ヘルスチェック
curl -s http://localhost:8000/health

# プロファイルの取得
curl -s http://localhost:8000/api/profile
```

## 6. テスト戦略

### テスト種別と実行タイミング

| 種別 | ディレクトリ | ツール | 実行タイミング |
|---|---|---|---|
| 単体テスト | `tests/unit/` | pytest + TestClient | push毎（CI自動） |
| E2Eテスト | `tests/e2e/` | Playwright | PR作成前（Claude自己検証 + CI） |
| インテグレーション | `tests/integration/` | pytest（モックなし） | ローカル手動のみ |

### テスト実行コマンド

```bash
# 単体テストのみ（CI相当）
uv run pytest tests/unit/ -v

# E2Eテスト（アプリ起動後に実行）
docker compose up -d
uv run pytest tests/e2e/ -v

# 全テスト（unit + e2e）
uv run pytest tests/unit/ tests/e2e/ -v
```

### 実装完了の定義（アジャイル：常に動く状態を保つ）

以下がすべて満たされて初めて「完了」とする：

1. **ACの全項目がチェック済み**であること
2. **`uv run pytest tests/unit/` が全件パス**すること
3. **E2Eで該当機能が画面から操作できる**こと（Claudeが自律検証）
4. **既存テストがリグレッションしていない**こと
5. **PRが作成されチケットにリンク**されていること

### Claude Code による自律E2E検証の手順

実装完了後、Claude Code は以下を自律的に行う：

1. `docker compose up -d` でアプリを起動
2. `uv run pytest tests/e2e/ -v` を実行
3. 失敗した場合は修正して再試行
4. 全件パスを確認してからPRを作成

### チケットACの標準テンプレート

新チケットのACは以下の形式で書く：

```
## Acceptance Criteria
- [ ] （ユーザー視点）何が操作・確認できるか
- [ ] どのAPIが呼ばれ何が返るか（バックエンドの場合）
- [ ] `uv run pytest tests/unit/` が全件パスすること
- [ ] E2Eで画面から操作できること
- [ ] エラーケースの扱い
```
