# TomorrowsMeal — コンセプト & ピッチ戦略

## コアバリュー（ブラさない前提）

1. **毎日3食の献立を提案する**
2. **インプット3種類を受け取る**
   - 自動取得情報（アレルギー・好き嫌い・調理器具・食の方向性・直近提案履歴）
   - 毎回入力（食材在庫写真・調理時間・気分）
   - 過去フィードバックから学習した好み
3. **フィードバックを次の提案に反映する**

---

## 解きたい課題（ペルソナで語る）

- 共働き世帯の「今日何作ろう」という**献立検討の認知負荷（名もなき家事）の削減**
- 冷蔵庫の余り食材を使い切る**食品ロス削減**
- 高齢者・単身者の**栄養管理の民主化**（専門知識なしで栄養バランスが整う）

## 測定するアウトカム（Output でなく Outcome / Impact）

- 食品ロス削減率（食材使い切り率）
- 栄養目標達成率
- 献立決定時間の短縮 / 調理時間削減

これらを **Four Keys 的な健康指標**として「献立版ダッシュボード」に実測値で可視化する。

## 審査員別ピッチ戦略

> 審査員4名のX/ブログ/登壇リサーチ結果に基づく。

### ばんくし王（確率的挙動の受容と信頼 / 事業・社会接続 / 90点の先の1点）

- 「提案の外れ（確率的挙動）を eval とFBで受容・信頼に変える設計にした」
- 「大規模モデルで出る90点の先、"あなた専用の1点"をどう作るか」

### 佐藤一憲（Augmentation over Automation / マルチモーダル / Live API）

- 「Augmentation over Automation。最終判断は人に委ね、AIは意図を汲んで選択肢を広げる」
- 「冷蔵庫写真という曖昧・マルチモーダル入力から意図を汲む」（＋Live APIの音声デモ）

### 宮田（生成AI → 保守AI / Output < Outcome < Impact）

- 「生成AIでなく"保守AI"として作った。作って終わりでなく、嗜好は変わり続けるので育てて回る構造にした」
- 「Output でなく Outcome（食品ロス◯%削減）」

### 佐藤将高（基盤の安定と実測主義 / Four Keys）

- 「AIが安定して動く基盤（嗜好スキルの再現性・回帰評価）を内蔵」
- 「継続改善は GitHub Actions → Slack → Issue自動起票 で仕組み化」
- 「効果は実測で示す」
