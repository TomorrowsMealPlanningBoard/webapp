## 0. コアバリュー（ブラさない前提）

1. **毎日3食の献立を提案する**
2. **インプット3種類を受け取る**
   - 自動取得情報（アレルギー・好き嫌い・調理器具・食の方向性・直近提案履歴）
   - 毎回入力（食材在庫写真・調理時間・気分）
   - 過去フィードバックから学習した好み
3. **フィードバックを次の提案に反映する**

このコアを軸に、以下の要件を「積み増す」形で受賞を狙う。

## 1. 機能要件・開発優先度（Tier別）

> 詳細な課題設定・アウトカム定義は [webapp/CONCEPT.md](webapp/CONCEPT.md) を参照。

### 🥇 Tier 1：核となる機能（テーマと必然性の担保）

- **① 献立エージェントの「自己改善ループ」（DevOps for AI Agent）** ← 最重要・テーマ直結
  - ユーザーFBによる「好みの学習」に留まらず、提案の質そのものを **LLM-as-judge** で自動評価し、ユーザー固有のルール（嗜好プロファイル）を継続的に改善・バージョン管理する。
  - 詳細は §4「2つのループ」。

- **② マルチモーダル冷蔵庫認識 ＋ ADKマルチエージェント構成**
  - Gemini Vision で冷蔵庫内の在庫（食材・量・鮮度/賞味期限の推定）を**構造化認識（Structured Outputs）**。
  - 単一の巨大プロンプトを避け、役割ごとに特化した4エージェントで構成（§3・§5）。重い処理は非同期実行。

- **③ 設計としての「Human-in-the-loop」の明示**
  - 完全自動化を避け、「AIの提案 → ユーザーの選択・微修正 → データの学習」を意図的に組み込む。
  - 非推奨要件の「自動発注」を扱う場合も全自動にせず、必ず「人が最終承認」する設計をピッチ・システム両方に明示（ガードレール）。

### 🥈 Tier 2：体験の飛躍（加点要素）

- **④ Gemini Live API による「調理中の音声インタラクション」**
  - 手が離せない調理中に「次どうする？」「玉ねぎ切らした、代わりある？」といった音声相談へリアルタイムに献立を組み替える。

- **⑤ 能動的（Proactive）な自律提案**
  - 「賞味期限が近い食材の優先」「前日の栄養バランス（Health API連携）に基づく調整」「カレンダー連携での作り置き提案」などを自律提案し、"エージェントである必然性"を可視化する。

### 🥉 Tier 3：ストーリーの鋭敏化

- §1 の課題設定とアウトカムの数値化（ダッシュボード）。

## 3. データ設計：データ特性に応じた3層の役割分離

> 「すべてを無差別にベクトル化して検索ノイズを増やす」ことを避け、データ特性ごとに処理方式を分ける。これが本システムの設計思想の中核。

| 層 | 内容 | 管理・処理方式 | 選定理由（審査員へのアプローチ） |
| --- | --- | --- | --- |
| **層1：安全・ハード制約** | アレルギー / 調理器具 / 禁止食材 | **決定的フィルタ**（構造化テキスト・`if`文による機械的除外）。**ベクトル検索は使わない** | ベクトル検索（確率的挙動）によるアレルギー食材の見落とし事故を防ぐ。ガードレールの構築。 |
| **層2：ユーザーの嗜好プロファイル** | FBから学習したルール・原則 | **構造化テキスト ＋ Git管理**（`CLAUDE.md` 的な「嗜好スキルファイル」） | 人間が読める・監査できる透明性。diff/PRレビュー、GitHub Actionsでの定期evalによる回帰評価を可能にするため。 |
| **層3：ユーザーFBの好み学習** | 自由記述FB（調理後コメント等） | **Agent Platform Memory Bank**（フルマネージド長期記憶・類似度検索内蔵） | ユーザーごとにスコープ分離された記憶を会話/FBから自動生成・自己整理。AlloyDB/pgvectorの自前運用（Auth Proxy・接続プール・都度destroy）を排除。 |
| **層3'：お気に入り外部レシピソース** | 外部URL（YouTube・ブログ）由来の要約 | **ベクトルDB不使用** — LLMで要約テキスト化し構造化DB（層2相当）に保存、生成時に**全件プロンプト直接注入** | 1ユーザーあたり数件〜数十件の小規模のため、ベクトル検索は過剰設計。RAG Engine(RagManagedDb)は常時起動で約$65/月かかり本用途には不釣り合いと判断し不採用（詳細は移行Epic参照）。 |

> **⚠️ 要検討事項（未確定）**: 層2を「ユーザーごとにGitで管理」する案は透明性・eval回帰の観点で強力だが、ユーザー数がスケールすると運用が重い。MVPでは「Git管理する対象は"エージェント共通の嗜好抽出ロジック/プロンプト"（＝ループB）」に絞り、個々のユーザー嗜好は **Agent Platform Memory Bank**（ユーザーごとにスコープ分離された長期記憶を会話/FBから自動生成・自己整理するフルマネージド機能）で管理する切り分けを推奨。ピッチでは「Git管理の透明性（＝ループB）」と「エージェントの記憶（＝ループA / Memory Bank）」を対で打ち出す。
>
> **※安全制約との分離（重要）**: Memory Bankは確率的な記憶であり、**層1（アレルギー・禁止食材）には絶対に使わない**。層1は決定的フィルタ（if文）を厳守する。この「確率的な記憶」と「決定的な安全ガードレール」の意図的な分離自体が設計の主張になる。

### フィードバックのデータ形式分離（層をまたぐ実装）

- **外部レシピソース（ベクトルデータ）:** テキストをベクトル化して格納。「この動画の味付けの方向性に近いレシピ」を引き出すセマンティック検索用。
- **フィードバック（構造化メタデータ）:** 「不採用タグ」「評価スコア」はスカラー値/配列（JSONB）で保存。ベクトル検索時の**ハードフィルタ（絶対的な除外・包含条件）**として利用。
- **フィードバック（自由記述テキスト）:** 「もう少し塩気が欲しかった」等はベクトル化し、過去の嗜好コンテキストとして保存。

**アーキテクチャ上の最適解:** 「外部レシピソースのベクトル」に近く、かつ「除外タグ（例：ナス、辛口）」を確実に排除する**ハイブリッド検索（ベクトル検索 ＋ メタデータフィルタリング）**を構成する。

## 4. 真のDevOpsを実現する「2つのループ」

システムが真の「DevOps × AI Agent」を満たすため、**製品機能としての学習ループ**と、**開発フローとしての継続改善ループ**を明確に分離・連携させる。ループAだけではDevOpsではない。ループBがあって初めて「DevOps × AI Agent」となる。

### ループA（製品のML学習ループ / 燃料）
ユーザーFBに基づき、層2の嗜好プロファイルを動的に更新し、次回の提案精度を向上させる**実行時の製品機能**（データフライホイール）。

### ループB（テーマが求める "DevOps" / 駆動輪）
ループAで蓄積された集約FBをもとに、開発者が**エージェント自体（プロンプト/ロジック）を継続改善するライフサイクル**。

#### 🔄 統合パイプラインのフロー
1. **バージョン管理:** 嗜好抽出ロジック/プロンプトをGitで管理し、変更をdiff/PRで可視化。
2. **定期eval:** GitHub Actions で、過去の提案と実際の評価を **LLM-as-judge** で回帰テスト。
3. **自動起票:** 品質スコア低下時に、プロンプト/ルールの修正案を含む**改善PRを自動起票**（＋Slack通知）。
4. **レビュー & デプロイ:** 人間のレビュー（Human-in-the-loop）を経てマージ → **Cloud Run へ自動デプロイ**。
5. **可観測性:** **Cloud Trace** で監視し、「提案品質スコア」の推移を実測値としてダッシュボードへ可視化。

## 5. 全体アーキテクチャ

### 5.1 アーキテクチャ図（ランタイム＝ループA）

```text
[フロントエンド] スマホブラウザ (React / Firebase Hosting でホスト)
       │ ▲
       │ │ (HTTPS / Generative UI[A2UI, 加点要素]: インタラクティブなレシピカード、FB用スマートチップ)
       ▼ │
[バックエンド] Cloud Run (Python / FastAPI / Webサーバー / UIレンダリング制御)
       │      ※ ADKの4エージェントはこのプロセス内に同居（マイクロサービス化しない）
       │
       ├─► [ADK Orchestrator: ループ・並列制御]
       │       │
       │       ├─► 1. Context Retriever Agent (プロファイル・FB取得)
       │       │        ├──► [層1/層2データストア] 構造化DB (Firestore 等 ※ベクトル不要)
       │       │        │        - 層1: 静的プロファイル/ハード制約 (アレルギー・器具・禁止食材) ← 決定的フィルタ
       │       │        │        - 層2: 構造化FB (不採用タグ等のメタデータ)             ← メタデータフィルタ
       │       │        ├──► [層3 好み学習] Agent Platform Memory Bank (フルマネージド長期記憶)
       │       │        │        - FB/会話からユーザーごとの好みを自動抽出・自己整理・類似度検索
       │       │        └──► [層3' 外部レシピソース] 構造化DB (層2相当・ベクトル不要)
       │       │                 - お気に入りURL(YouTube/ブログ)由来の構造化データを全件プロンプト直接注入
       │       │
       │       ├─► 2. Vision Analyzer Agent (食材解析)
       │       │        └──► [LLM] Gemini Enterprise Agent Platform (Gemini + Structured Outputs)
       │       │
       │       ├─► 3. Recipe Generator Agent (レシピ生成 / 3案)
       │       │        └──► [LLM] Gemini Enterprise Agent Platform (Gemini)
       │       │
       │       └─► 4. Recipe Reviewer Agent (制約監査 / ガードレール)
       │                └──► [LLM] Gemini Enterprise Agent Platform (Gemini)
       │
       └─► [MCP サーバー群 / 外部API]
               ├─► (Tier2) Gemini Live API (調理中の音声インタラクション)
               ├─► (拡張) Google Health API
               └─► Web Scraper (YouTube / ブログ記事取得 → 層3'の構造化DBへ)
```

> **注**: Gemini のモデル世代は開発時点の最新（Gemini 3 世代 Flash/Pro 相当）を採用。バージョンは実装時に確定する。
>
> **注（プラットフォーム名称）**: 2026 Next で **Vertex AI は「Gemini Enterprise Agent Platform」に改称**（Agentspace 等を統合）。本SPEC内の「Gemini Enterprise Agent Platform」は旧 Vertex AI を指す。好み学習の基盤は AlloyDB(pgvector) の自前運用をやめ、同プラットフォームのフルマネージド機能 **Memory Bank（層3の好み学習）** に移行する。外部レシピソース（層3'）はベクトルDBを使わず構造化DB＋プロンプト直接注入で対応する（§5.4、RAG Engineは検討の上不採用）。

### 5.2 マルチエージェントによる処理フロー（提案時）

ADKのワークフロー制御を活用し、**生成と監査のループ**を構築する。

1. **データ収集フェーズ（並列処理）**
   - **Context Retriever Agent** がハイブリッド検索（層1/層2の構造化DBからメタデータフィルタでアレルギー・NG食材を決定的に除外 ＋ Memory Bank のベクトル類似度検索で好みを抽出 ＋ 層3'の構造化DBから外部レシピソースを全件取得）を行い、プロファイルとコンテキストを構築。
   - 同時に **Vision Analyzer Agent** が冷蔵庫画像を Gemini の Structured Outputs で解析し、使用可能な食材と状態のJSONリストを出力。

2. **生成フェーズ**
   - **Recipe Generator Agent** が上記2エージェントの出力＋ユーザーの現在の気分・許容調理時間を統合し、**3つのレシピ案**（手順含む）を生成。

3. **監査・承認フェーズ（ループ処理 / ガードレール）**
   - **Recipe Reviewer Agent** が3案を検査：アレルギー物質の混入、除外指定タグ（食材・味付け）の含有、未所持の調理器具の使用（例: オーブンがないのにオーブン焼き）がないか厳格チェック。
   - **【リジェクト時】** 理由を添えて Generator に差し戻し、該当レシピを再生成（ADKのLoop制御）。
   - **【承認時】** すべての制約をクリアしたらフロントエンドへレスポンス。

4. **UI描画フェーズ**
   - Cloud Run が承認済みレシピを「レシピカード」として描画。各カードには即座にFB可能な「不採用（もう表示しない）」ボタンを配置。
   - Generative UI（A2UI）は加点要素。**コア動作は通常のReact描画でも成立する**設計とし、実装リスクをヘッジする。

### 5.3 フィードバックループのUX/データフロー（＝ループA の実装）

> 設計思想: **「簡単にFBできること」と「LLMが理解しやすい構造化データを集めること」を両立**させる。FBしてもらうことが最重要なので、ユーザーの入力摩擦を最小化する。

1. **提案時（ネガティブFB）**
   - 3案それぞれに「不採用（もう表示しない）」ボタンを配置。
   - 押下時、料理名だけでなくレシピの**特徴タグ（例: #揚げ物, #豚肉, #辛い）を内部抽出**し、層2の構造化DBに「除外タグ（Negative Tags / 絶対的フィルタ）」として蓄積。
   - 次回以降、Context Retriever Agent がこれを即座に拾う。料理名そのものより「不採用の理由（属性）」を学習させる方が提案精度が劇的に向上する。

2. **調理後（5段階＋詳細FB）**
   - まず**星タップの5段階評価**のみ（自由記述は強制しない＝FB率低下を防ぐ）。
   - 星タップ直後、**動的にスマートチップ（選択式タグ）**をインライン表示：
     - 星1〜2 → 「工程が大変だった」「味が合わなかった」「量が多かった」
     - 星4〜5 → 「味付けが最高」「手軽だった」「子供が喜んだ」
   - 1タップで選択 → 完全に構造化されたデータ（メタデータ）として回収。
   - 自由記述欄はその下に「その他、気づいた点があれば」とオプショナル配置 → ベクトル化して層3へ。

### 5.4 お気に入りレシピソース（外部URL）の統合

最初はデータがないのでゼロベース生成。ユーザーが好みのソース（YouTube動画・ブログ）を提供できれば、レシピ生成の**シード（種）**として機能させる。

- **URL入力時:** バックエンドでスクレイピング（YouTubeなら動画タイトル・説明文・字幕を取得）。
- **構造化と保存:** LLM（Source Extractor Agent、`prompts/source_extraction.md`）で以下のフィールドに**分けて（自由記述の1つの要約テキストにせず）**抽出し、構造化DB（層2相当、ベクトル化しない）にJSON等の構造化形式で保存する。
  - `seasoning_tendency`（味付けの傾向。例: 「醤油とみりんベースの甘辛い味付けを好む傾向」）
  - `favorite_ingredient_combos`（好まれる食材の組み合わせ。配列）
  - `cooking_style`（調理スタイル。例: 「短時間で作れる炒め物中心」）
  - `tags`（検索・分類用の短いタグ。配列）
  - `source_title` / `source_url`（出典URL・タイトル。抽出結果に含めず、保存時にアプリ側で付与。ユーザーへの表示・監査用）
- **生成時への反映:** Recipe Generator 実行時に、ユーザーが登録した構造化データを**全件そのままプロンプトへ直接注入**する（フィールドごとに整形して「ユーザーは過去にこのスタイルを好んでいます。今回の食材でこのアプローチに類似したレシピを構築してください」）。構造化しておくことで、後から特定フィールドだけを層1/層2のフィルタ条件と組み合わせたり、Reviewer Agentの監査対象にすることも容易になる。

> **設計判断（ベクトルDB不採用）**: お気に入りURLは1ユーザーあたり数件〜数十件程度の小規模であり、SPEC §3で示した「1ユーザーの小規模履歴はコンテキストに直接注入し過剰設計を回避」の方針をそのまま適用する。件数が実運用で数百件を超えるようなら、その時点で RAG Engine 等のベクトル検索導入を再検討する（現時点では時期尚早と判断）。

## 6. デプロイ構成：Cloud Run に集約する（Agent Runtime ではない）

### 6.1 決定的な要因：Generative UI（A2UI）のサポート可否

| デプロイ先 | UI表現能力 |
| --- | --- |
| **Agent Runtime (Agent Engine)** | ADKを最も素直にホストできるが、**A2UI不可（テキストのみ）**。インタラクティブなレシピカードやFB用スマートチップを動的描画できない。 |
| **Cloud Run** | 汎用コンテナ環境。**A2A / A2UI をネイティブに実装・ホスト可能**。JSON Lines でストリーム配信しリッチUIを描画できる唯一の選択肢。 |

→ **本プロジェクトは Cloud Run に統一する**（CLAUDE.mdの確定方針とも一致）。

### 6.2 リソース割り当て：必要な Cloud Run は「1つ」

| 役割 | 実体・デプロイ先 | Cloud Runの数 | 理由 |
| --- | --- | --- | --- |
| **フロントエンド** (React等) | **Firebase Hosting**（静的ホスティング） | **0個** | ビルド後は静的ファイル。高価な計算資源であるCloud Runは不要。コストほぼゼロで世界中に高速配信。 |
| **バックエンド** (API / UI制御) | **Cloud Run** (Python) | **1個** | リクエストを受付、UIフォーマットのレスポンスを制御するWebサーバー。 |
| **4つのエージェント** (ADK) | **バックエンドに内包** | **0個** | ADK仕様上、4エージェントは1つのPythonプログラム内の「モジュール」として記述。個別にサーバーを分けない。 |

### 6.3 エージェントごとに Cloud Run を分けない理由（マイクロサービスの罠）

MVP段階で4エージェントを別々のCloud Runに分割すると：
1. **ネットワーク遅延の致命的増大:** 「生成→監査→差し戻し（Loop）」のたびにCloud Run間HTTPS通信が発生。LLM推論時間に通信遅延が重なり実用速度に達しない。
2. **セッション・状態管理の複雑化:** 「どのユーザーのどの冷蔵庫画像か」というコンテキストをネットワーク越しに引き回す実装が必要になり開発難易度が跳ね上がる。1プロセス内なら引数の受け渡しで完結。
3. **コストの無駄:** 4コンテナが個別にコールドスタートし、それぞれ課金される。

→ **「フロントは静的ホスティングで0個、バックエンドとエージェント群は1つのCloud Runに集約」**が性能・コスト・開発速度すべてでベストプラクティス。

### 6.4 Cloud Run 実装ベストプラクティス

1. **ADKロジックをバックエンドAPIに完全内包:** 4エージェントの協調動作を Cloud Run 上の1つのPython（FastAPI等）プロセス内で完結させ、ネットワーク遅延のない高速ループを実現。
2. **A2A/A2UI実装の簡略化（加点要素採用時）:** `to_a2a()` 等のユーティリティを活用し、`DataPart` の `mimeType` に `application/json+a2ui` を宣言する実装を標準化。
3. **マネージドデータ基盤への接続:** Memory Bank は Google API 経由（`VertexAiMemoryBankService` 等）でアクセスするため、AlloyDB Auth Proxy のようなサイドカーや接続プールの自前運用は不要。認証は Cloud Run 実行サービスアカウントの IAM（ADC）で完結し、Secret Manager でのDBパスワード管理も撤廃する。層1/層2/層3'の構造化DB（Firestore 等）も同様にサーバーレスで接続する。
