# TomorrowsMeal — デモ動画 台本・絵コンテ（Proto Pedia提出用）

> **用途**: ハッカソン提出に必須のデモ動画（YouTube/Vimeo）を、撮影・録音・字幕・編集・サムネまでこのまま実行できる完全台本。
> **推奨尺**: **2:40（ファイナリスト・カット / 本命）** ／ フル版 3:00。3分を超えないこと（審査員は多数の動画を連続視聴する。冗長さは即減点）。
> **録画対象URL**: https://tomorrows-meal-webapp-td7ugk7iva-an.a.run.app （＋ GitHub Actions / PR 画面）
> **画面比率**: 動画は **16:9 横（1920×1080）** で書き出し。アプリはモバイル縦（幅 390px 想定）を画面中央に置き、左右の余白にキーワード字幕とアーキ図を差し込む「縦アプリ＋横フレーム」構成。
> **参照**: [PITCH.md](../PITCH.md)（審査員の刺しどころ）/ [SPEC.md](../SPEC.md)（2ループ・3層設計）/ [docs/hackathon-overview/rules.md](hackathon-overview/rules.md)（審査5項目）/ [docs/hackathon-overview/judges.md](hackathon-overview/judges.md)（審査員5名）

## 0. この改訂で「勝ちにいく」ための方針（旧版からの変更点と根拠）

旧版は「審査5項目 × 審査員を全部触る」網羅性は高かったが、**特徴量（feature）の列挙**に寄っており、①最初の勝負どころ（冒頭3秒）の実弾セリフがない、②Proto Pedia必須の**アーキテクチャ図を動画で見せる導線**がない、③審査員5人目（古久保／Elasticsearch）が未対応、④ダッシュボードが**デモ用シードデータ**である事実を伏せると「数字を作った」疑念で逆に減点され得る、⑤3:20と尺オーバー、という弱点があった。本版はそれを全て潰す。

**中心に据える1つの主張（これだけは絶対にブレさせない）**:
> 「大規模モデルの“90点の提案”を、あなたのフィードバックで“あなた専用の91点”に育て（ループA）、そのエージェント自身をevalとPRで開発者が磨き続ける（ループB）。**作って終わらない、保守されるAIエージェント。**」

この1文が S1 のフックと S8 のクロージングで対になって回収される構造にする。他のシーンは全てこの主張の「証拠」として配置する。

### 審査5項目 × 主要シーン（配分）

| 審査基準 | 主に触れるシーン |
| --- | --- |
| ① AIエージェントが価値の中心か | S2（4エージェント・生成→監査Loop）、S3（好み学習）、S5（能動提案） |
| ② 課題へのアプローチ力 | S1（フック：名もなき家事の認知負荷）、S8（クロージング） |
| ③ ユーザビリティ | S2（写真1枚＋気分入力）、S3（1タップFB／スマートチップ）、S4（音声ハンズフリー） |
| ④ 実用性・体験価値 | S4（調理中の音声相談）、S5（能動提案）、S6（アウトカム実測） |
| ⑤ 実装力 | S2（ADK 4エージェント・決定的フィルタ）、S6b（アーキ図・設計判断）、S7（ループB: eval→PR→デプロイ） |

### 審査員5名を「どのシーンで刺すか」（PITCH.md・judges.md 準拠）

- **ばんくし王**（確率的挙動の受容と信頼 / 90点の先の1点）: **S3（育つAI）**＋**S7（eval）**。「提案の外れをevalとFBで信頼に変える」「90点の先の“あなた専用の1点”」を明言。
- **佐藤一憲**（Augmentation over Automation / マルチモーダル / Live API）: **S2（冷蔵庫写真＝曖昧なマルチモーダル入力）**＋**S4（Gemini Live 音声）**。「最終判断は人、AIは選択肢を広げる」を明言。
- **宮田**（生成AI→保守AI / Output<Outcome<Impact）: **S3（好み学習ループA）**＋**S6（アウトカム実測）**。「作って終わりでなく、嗜好は変わり続けるので育てて回る保守AI」「OutputでなくOutcome」を明言。
- **佐藤将高**（基盤の安定・実測主義 / Four Keys）: **S7（ループB: GitHub Actions eval→改善PR自動起票＋Slack `#quality-watch` 通知→Cloud Run自動デプロイ）**＋**S6（実測ダッシュボード）**。「継続改善を仕組み化」「異常はSlackで検知」「効果は実測」を明言。
- **古久保武雄**（Elasticsearch / Principal Solutions Architect＝アーキテクチャ選定の妥当性）: **S6b（設計判断カット）**。**ベクトルDBを“あえて使わない”設計判断**（1ユーザー数件規模ではプロンプト直接注入が適切／RAG Engineは常時起動$65/月で過剰）を根拠つきで語り、Memory Bankとの役割分担を示す。「必要な所にだけ適切な技術を置く」アーキテクトの目線に刺す。※Elasticsearch自体は本作では不使用のため、**使っているフリはしない**（虚偽は最も危険）。技術選定の“納得度”（審査基準⑤の文言）で勝負する。

### トーン
優しく親しみやすい生活者目線から入り、後半で技術的必然性（ADK / LLM-as-judge / Memory Bank / 決定的フィルタ / Human-in-the-loop / Outcome）を織り込む。誇張せず、画面で実際に動くものだけを見せる。専門用語は字幕に文字で残す（音声オフ視聴と実装力評価の両取り）。

## 1. 動画全体の設計方針

### 冒頭10秒のフック（最重要 / 離脱防止）
審査員は大量の動画を連続で見る。**最初の10秒で「痛み（献立を考える認知負荷）」と「解決の予感」を映像だけで見せる**。無音でも意味が通るように、映像・テロップ・SEで殴る。素材は **Seedance 2.0で生成する高品質な実写風映像**（4カット構成）を使い、生活者への共感で掴む。
- 0:00–0:03 30歳の主婦が日本の家庭の冷蔵庫を開け、中をのぞき込む（無言、生活音のみ）
- 0:03–0:06 中身のインサート（日本のスーパーで買える食材＝肉・魚・野菜がバランスよく）＋彼女の思案顔。大テロップ「今日、何作ろう？」
- 0:06–0:08 小さくため息。「栄養・在庫・好み・重複…を毎日1日3回」の負荷が伝わる表情
- 0:08–0:10 手元のスマホに3案が「ポン」と並び、表情が明るくなる（痛み→解決の対比）。テロップ「AIエージェントが、あなた専用に。」
- この10秒に**ナレーションを被せない**（映像だけで殴る）。ナレーション開始は0:10から。
- カット割り・登場人物・食材・生成プロンプトの詳細は [seedance-prompt.md](seedance-prompt.md) を参照。**登場人物は主婦1名のみ**（家族・子供は出さない）。

### タイトルカード & エンドカード（必須要件の可視化）
- **オープニング（0:10付近、2秒）**: アプリ名ロゴ＋一行コピー「冷蔵庫の残り物から、あなた専用の献立を。」＋画面下部に**技術ストリップ**を常時 or 冒頭表示：`Cloud Run ／ Gemini (Vertex) ／ ADK ／ Gemini Live API ／ GitHub Actions`。→ ハッカソン必須要件（Cloud Run＋Google Cloud AI）を審査員が0秒で確認できる。
- **エンドカード（末尾5秒）**: アプリ名／一行コピー／**動作URL**／GitHub URL／`#findy_hackathon`／チーム名。Proto Pediaの3提出物（リポジトリ・デプロイURL・作品）と符合させ、審査導線を閉じる。

### 尺配分（2つのカットを用意）

| シーン | フル版(3:00) | ファイナリスト版(2:40 / 本命) |
| --- | --- | --- |
| S1 フック（冒頭10s Seedance）＋課題 | 18s | 18s |
| S2 コア体験（写真→3案・監査Loop） | 42s | 40s |
| S3 育つAI（ループA）【最大の見せ場】 | 33s | 32s |
| S4 音声相談（Gemini Live） | 20s | 16s |
| S5 能動提案 | 14s | 7s |
| S6 アウトカム実測 | 18s | 15s |
| S6b アーキ図＋設計判断 | 15s | 12s |
| S7 ループB（eval→PR→デプロイ）【差別化】 | 28s | 26s |
| S8 クロージング＋エンドカード | 14s | 14s |
| **合計** | **202s (3:22)** | **180s (3:00)** |

> 注: **冒頭フックは離脱防止の最重要ポイントのため、Seedance 2.0の高品質映像に10秒割り当てる**（ファイナリスト版でもS1は削らず18s固定）。その分の帳尻はS5の圧縮（10s→7s：能動提案カードを1枚に絞る）で合わせ、**ファイナリスト版=ちょうど3:00**を維持する。フル版の実合計は3:22なので、**3分厳守**のためS5→S4後半→S6bの順に“間”を詰める（ナレーションは削らない）。さらに攻めるなら S6bを字幕オーバーレイのみ（ナレなし6秒）にして2:40も可能。

### 削ってよい順（尺オーバー時）
S5 → S4後半 → S6b → S6。**S1冒頭フック（Seedance 10s）・S2・S3・S7は削らない**（フックは離脱防止、S2・S3・S7は審査5項目と審査員5名の刺しどころが集中）。

| 尺 | 42s |

## 3. ナレーション全文（録音用・通し読み）

> 目安の尺つき。句読点で軽く息継ぎ。専門用語（太字箇所）は少しゆっくり。**S1冒頭10秒は無音**（Seedance生成映像とテロップのみ）、ナレーションは0:10から。

**【S1｜約18秒 / うち先頭10秒無音（Seedanceフック）】**
（無音／フック映像）…毎日やってくる「今日、何作ろう」。栄養も、在庫も、好みも、昨日と被らないかも——限られた時間の中で同時に考える。しかも1日3回。この“名もなき家事”の認知負荷を減らしたい。TomorrowsMealは、冷蔵庫の残り物から、あなた専用の献立を**AIエージェント**が提案するアプリです。

**【S2｜約42秒】**
使い方はシンプル。冷蔵庫の写真を1枚。**Gemini Vision**が食材と鮮度を構造化して読み取ります。あとは、今日の気分と使える時間を選ぶだけ。すると**ADK**で組んだ4つのエージェント——文脈収集・画像解析・レシピ生成、そして「監査」が動きます。生成されたレシピは、アレルギーや苦手食材、持っていない調理器具が混ざっていないかを**監査エージェント**が厳格チェックし、問題があれば生成側へ差し戻して作り直す。この**生成と監査のループ**で、安心して選べる3案だけが届きます。

**【S3｜約33秒】**
ここが一番の肝です。気に入らなければ「不採用」を1タップ。すると料理名ではなく“揚げ物”“辛い”といった**特徴タグ**を抽出して、確実に除外する条件として学習します。作った後は星と、その場で出るチップを選ぶだけ。書く手間はほぼゼロなのに、構造化されたフィードバックが貯まる。自由なひとことは、**Memory Bank**がユーザーごとの長期記憶として自動で整理。これが提案を回すたびに賢くなる“燃料”、**ループA**です。大きなモデルが出す90点の提案の、その先の“**あなた専用の1点**”を作りにいきます。

**【S4｜約20秒】**
調理は手が離せません。だから声で相談できる。「玉ねぎを切らした、代わりは？」——**Gemini Live API**でリアルタイムに会話し、その場で代わりの食材を提案。しかもアレルギーや苦手食材は必ず避けたうえで。AIが勝手に決めるのではなく、最終判断は人に委ね、選択肢を広げる。**Augmentation over Automation**の考え方です。

**【S5｜約14秒】**
待っているだけではありません。賞味期限が近い食材を先に使い切る献立や、直近の栄養バランスを踏まえた調整を、AIの側から能動的に提案します。指示を待つ道具ではなく、先回りする——“**エージェントである必然性**”がここにあります。

**【S6｜約18秒】**
そして、成果は**実測**で示します。食品ロス削減率、栄養目標の達成率、献立を決めるまでの時間。作った機能の数ではなく、生活がどう変わったかという**Outcome**を可視化する。さらに、提案品質そのもののスコアも時系列で追いかけます。

**【S6b｜約15秒】**
構成はこの1枚に。**Cloud Run**上で**ADK**の4エージェントが動き、安全に関わる層1はベクトル検索を使わず**決定的なif文フィルタ**で。一方、好み学習は**Memory Bank**に任せる。確率的な記憶と、決定的な安全ガードレールを意図的に分離しています。好みのソースは1人あたり数件規模なので、ベクトルDBはあえて使わずプロンプトに直接注入——必要な所に、必要な技術だけを置きました。

**【S7｜約28秒】**
これは**DevOps × AI Agent**のハッカソン。だから“作って終わり”にしません。**GitHub Actions**が毎日、過去の提案を**LLM-as-judge**で自動評価。スコアが下がると、プロンプトの改善案を載せた**改善PRを自動で起票**し、Slackの `#quality-watch` に通知します。ただし自動マージはしない。**Draft PR**として人がレビューし、承認して初めてマージ、そして**Cloud Runへ自動デプロイ**。**Human-in-the-loop**を効かせて、エージェント自身を継続改善する仕組み——これが**ループB**です。

**【S8｜約14秒】**
TomorrowsMealは、あなたのフィードバックで育つ**ループA**と、開発者がエージェントを磨き続ける**ループB**。2つのループで“あなた専用の献立”を毎日つくる、**保守されるAIエージェント**です。今日の「何作ろう」を、もう悩まない。
