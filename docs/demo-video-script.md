## 1. 動画全体の設計方針

### 3分で審査基準5項目を全部触る配分

| 審査基準 | 主に触れるシーン |
| --- | --- |
| ① AIエージェントが価値の中心か | S2（4エージェント・生成→監査Loop）、S3（好み学習）、S5（能動提案） |
| ② 課題へのアプローチ力 | S1（導入：今日何作ろうの認知負荷）、S6（クロージング） |
| ③ ユーザビリティ | S2（写真1枚＋気分入力）、S3（1タップFB／スマートチップ）、S4（音声） |
| ④ 実用性・体験価値 | S4（調理中の音声相談）、S5（能動提案）、S6（アウトカム実測） |
| ⑤ 実装力 | S2（ADK 4エージェント・決定的フィルタ）、S7（ループB: eval→PR→デプロイ） |

### 審査員4名を「どのシーンで刺すか」

- **ばんくし王**（確率的挙動の受容と信頼 / 90点の先の1点）: **S3（育つAI）**＋**S7（eval）**。「提案の外れをevalとFBで信頼に変える」「大規模モデルの90点の先、あなた専用の1点」を明言。
- **佐藤一憲**（Augmentation over Automation / マルチモーダル / Live API）: **S2（冷蔵庫写真＝曖昧なマルチモーダル入力）**＋**S4（Live API音声）**。「最終判断は人、AIは選択肢を広げる」を明言。
- **宮田**（生成AI→保守AI / Output<Outcome<Impact）: **S3（好み学習ループA）**＋**S6（アウトカム実測）**。「作って終わりでなく、嗜好は変わり続けるので育てて回る保守AI」「OutputでなくOutcome」を明言。
- **佐藤将高**（基盤の安定・実測主義 / Four Keys）: **S7（ループB: GitHub Actions eval→改善PR自動起票→Cloud Run自動デプロイ）**＋**S6（実測ダッシュボード）**。「継続改善を仕組み化」「効果は実測」を明言。

### トーン
優しく親しみやすい生活者目線から入り、後半で技術的必然性（ADK / LLM-as-judge / Memory Bank / 決定的フィルタ / Human-in-the-loop / Outcome）を織り込む。誇張せず、画面で実際に動くものだけを見せる。

## 3. ナレーション全文（録音用・通し読み）

> 目安の尺つき。句読点で軽く息継ぎ。専門用語（太字箇所）は少しゆっくり。

**【S1｜約20秒】**
毎日やってくる「今日、何作ろう」。冷蔵庫を開けても献立が決まらない。この“名もなき家事”の認知負荷を減らしたい。TomorrowsMealは、冷蔵庫の残り物から、あなた専用の献立を**AIエージェント**が提案するアプリです。

**【S2｜約45秒】**
使い方はシンプル。冷蔵庫の写真を1枚。**Gemini Vision**が食材と鮮度を構造化して読み取ります。あとは、今日の気分と使える時間を選ぶだけ。すると、**ADK**で組んだ4つのエージェント——文脈収集・画像解析・レシピ生成、そして「監査」が動きます。生成されたレシピは、アレルギーや苦手食材、持っていない調理器具が混ざっていないかを**監査エージェント**が厳格チェックし、問題があれば生成側へ差し戻して作り直す。この**生成と監査のループ**で、安心して選べる3案だけが手元に届きます。

**【S3｜約35秒】**
ここが一番の肝です。気に入らなければ「不採用」を1タップ。すると料理名ではなく“揚げ物”“辛い”といった特徴タグを抽出して、確実に除外する条件として学習します。作った後は星と、その場で出るチップを選ぶだけ。書く手間はほぼゼロなのに、構造化されたフィードバックが貯まります。自由なひとことは、**Memory Bank**がユーザーごとの長期記憶として自動で整理。これが提案を回すたびに賢くなる“燃料”、**ループA**です。大きなモデルが出す90点の提案の、その先の“あなた専用の1点”を作りにいきます。

**【S4｜約20秒】**
調理は手が離せません。だから声で相談できます。「玉ねぎを切らした、代わりは？」——**Gemini Live API**でリアルタイムに会話し、その場で代わりの食材を提案する。しかもアレルギーや苦手食材は必ず避けたうえで。AIが勝手に献立を決めるのではなく、最終判断は人に委ねて、選択肢を広げる。**Augmentation over Automation**の考え方です。

**【S5｜約15秒】**
待っているだけではありません。賞味期限が近い食材を先に使い切る献立や、直近の栄養バランスを踏まえた調整を、AIの側から能動的に提案します。“エージェントである必然性”がここにあります。

**【S6｜約20秒】**
そして、成果は実測で示します。食品ロス削減率、栄養目標の達成率、献立を決めるまでの時間。作った機能の数ではなく、生活がどう変わったかという**Outcome**を可視化する。さらに、提案品質そのもののスコアも時系列で追いかけます。

**【S7｜約30秒】**
これはDevOps × AI Agentのハッカソン。だから“作って終わり”にしません。**GitHub Actions**が毎日、過去の提案を**LLM-as-judge**で自動評価。スコアが下がると、プロンプトの改善案を載せた**改善PRを自動で起票**します。ただし自動マージはしません。Draft PRとして人がレビューし、承認して初めてマージ、そして**Cloud Runへ自動デプロイ**。**Human-in-the-loop**を効かせた、エージェント自身を継続改善する仕組み——これが**ループB**です。

**【S8｜約15秒】**
TomorrowsMealは、あなたのフィードバックで育つループAと、開発者がエージェントを磨き続けるループB。2つのループで“あなた専用の献立”を毎日つくる、保守されるAIエージェントです。今日の「何作ろう」を、もう悩まない。

## 5. 撮影のコツ

- **ローディングの間延び対策**: S2 の提案生成・S4 の音声応答は待ち時間が出やすい。①`min-instances=1`で温めておく、②編集でローディングを1.5〜2倍速にカット（ナレーションは等速）、③「AIが献立を考えています…」のローディング表示は“待つ楽しさ”として2〜3秒だけ意図的に見せ、残りはカット。
- **失敗時のリテイク方針**:
