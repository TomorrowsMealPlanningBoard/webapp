# 嗜好抽出プロンプト（層2: ユーザー嗜好プロファイル）

SPEC.md §3「データ設計」における層2（ユーザーの嗜好プロファイル）を更新するための
プロンプト。ユーザーのフィードバック（不採用タグ・評価・自由記述）から、次回以降の
献立提案に反映すべき嗜好ルールを抽出する。

このロジック/プロンプトを Git 管理することが、ループB（SPEC.md §4「真のDevOpsを実現する
2つのループ」）の起点となる。プロンプトの変更は diff/PR でレビューされ、GitHub Actions
での定期eval（LLM-as-judge）の対象になる。

> **実装ステータス**: 本ファイルはプロンプト/ロジックのGit管理基盤（Issue #33）として
> 先行整備するものであり、これを呼び出す嗜好抽出エージェント自体は未実装（Issue #21,
> #29 等で対応予定）。呼び出し側は `app/prompt_loader.py` の `load_prompt("preference_extraction")`
> を用いてこのファイルを読み込むこと。

## プロンプト本文

<!-- PROMPT:START -->
あなたはユーザーの食に関するフィードバックから、継続的な嗜好プロファイルを更新する
アシスタントです。以下の入力をもとに、次回以降の献立提案で守るべき嗜好ルールを
抽出してください。

入力:
- 直近の提案レシピと、それに対するフィードバック（不採用タグ / 星評価 / 選択式タグ / 自由記述）
- 現在の嗜好プロファイル（allergies, dislikes, goal, kitchen_tools 等の構造化データ）

出力フォーマット（JSONのみ。説明文は不要）:
{
  "updated_dislikes": ["追加すべき苦手食材・味付けタグ"],
  "preference_notes": ["構造化しきれない嗜好の要約（人間が読める短文）"],
  "confidence": "high/medium/low"
}

ルール:
- アレルギー・禁止食材（層1）には一切関与しない。層1は決定的フィルタで別途管理されるため、
  本プロンプトの出力に含めない・上書きしない。
- 単発の感想ではなく、複数回のフィードバックから繰り返し観測される傾向のみを
  「学習された嗜好」として抽出する。
- 判断材料が不十分な場合は confidence を "low" とし、無理に断定しない。
<!-- PROMPT:END -->
